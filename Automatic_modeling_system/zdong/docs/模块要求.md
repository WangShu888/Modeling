# 建筑自动建模模块要求文档

版本：`v0.2`

日期：`2026-03-30`

适用范围：`基于 CAD/PDF 图纸 + 文本需求生成 IFC，并由用户使用 Revit 查看结果`

## 1. 文档目标

本文档用于细化建筑自动建模系统中各个模块的职责、依赖、输入输出以及实施方式，作为后续需求拆分、开发实现和联调验收的依据。

本文档遵循当前系统主链路：

`图纸、文字信息 -> CAD/PDF解析 -> AI需求转化 -> 规则引擎 -> 参数化建模/执行计划 -> BIM引擎 -> 系统自检 -> 输出IFC文件 -> 用户使用Revit查看结果`

## 2. 适用前提

- 当前版本仅处理 `CAD/DWG/DXF/PDF` 图纸
- `DWG` 优先转换为 `DXF` 后解析
- 最终交付主格式为 `IFC`
- `Revit` 的角色是最终查看与使用，不承担系统内部验证职责
- 系统内部必须先完成自检，再允许导出

## 3. 模块总览

系统包含以下模块：

- 图纸、文字信息接入
- CAD/PDF 解析
- AI 需求转化
- 规则引擎
- 参数化建模 / 执行计划
- BIM 引擎
- 系统自检
- 自检不通过回退
- 输出 IFC 文件
- Revit 查看与使用

## 4. 模块详细要求

### 4.1 图纸、文字信息接入

#### 模块目标

统一接收用户上传的图纸和文本要求，并为后续解析与建模建立项目上下文。

#### 需要什么

- 用户上传入口
- 项目基础信息表单
- 文件存储服务
- 元数据管理

建议技术：

- `React`
- `FastAPI`
- `PostgreSQL`
- `MinIO / S3`

#### 输入

- `CAD/DWG/DXF/PDF` 图纸
- 用户文字需求
- 项目类型、地区、楼层、输出格式等表单信息

#### 输出

- `SourceBundle`
- `project_id`
- `request_id`
- `asset_id[]`
- `version_id`

#### 该怎么做

1. 用户上传图纸文件并填写项目基础参数。
2. 系统校验文件格式、大小、命名和重复上传情况。
3. 系统将原始文件存入对象存储，并在数据库中记录元数据。
4. 系统生成统一输入包 `SourceBundle`，供解析模块使用。

#### 实施要点

- 所有上传文件要保存原始版本
- 文件元数据必须记录上传时间、来源、版本关系
- 文本输入与图纸文件必须绑定同一个 `request_id`

### 4.2 CAD/PDF 解析

#### 模块目标

将图纸内容转换为程序可消费的几何与语义中间结果。

#### 需要什么

- `DWG -> DXF` 标准化工具
- `DXF` 解析器
- `PDF` 矢量内容解析器
- 可选 OCR 能力
- 图层映射规则

建议技术：

- `DWG TrueView`
- `ezdxf`
- `PyMuPDF`
- 可选 `Tesseract OCR`

#### 输入

- `DWG/DXF/PDF`
- 项目默认单位
- 图层映射配置

#### 输出

- `ParsedDrawingModel`
- `LayerMap`
- `GridMap`
- `DimensionEntities`
- `TextAnnotations`
- `StoreyCandidates`

#### 该怎么做

1. 若输入为 `DWG`，先转换为 `DXF`。
2. 解析 `LINE / POLYLINE / INSERT / TEXT / MTEXT / DIMENSION / LAYER`。
3. 从 `PDF` 中提取文字、线框、块状区域和坐标信息。
4. 识别轴网、墙线、门窗块、房间边界、标高和尺寸标注。
5. 统一单位、原点、朝向和楼层标高。
6. 输出统一的 `ParsedDrawingModel`。

#### 实施要点

- 必须处理单位归一化
- 必须记录图纸原点和建筑北向
- 必须区分矢量 `PDF` 与扫描 `PDF`
- 无法识别的图元要进入待确认清单

### 4.3 AI 需求转化

#### 模块目标

将文本要求和图纸解析结果转为结构化建模指令。

#### 需要什么

- 支持结构化输出的主控模型
- 固定 `JSON Schema`
- 提示词模板
- 字段补全策略

建议技术：

- `GPT-5.4`
- `GPT-5.4-nano`
- `Structured Outputs`
- `Function Calling`

#### 输入

- 用户文本需求
- `ParsedDrawingModel`
- 表单字段
- 项目默认参数

#### 输出

- `DesignIntent`
- `MissingFields`
- `Assumptions`
- `CompletionTrace`
- `ModelPatch`
- `ElementSelector`

#### 该怎么做

1. 将文本需求、图纸中间结果和表单字段一起送入模型。
2. 要求模型严格按 `JSON Schema` 输出。
3. 区分用户明确要求、系统默认值和 `AI` 补全项。
4. 对构件级修改生成筛选器和动作对象。
5. 输出结构化结果后再交给规则引擎处理。

#### 实施要点

- 不允许模型直接输出自由文本作为最终结果
- 所有自动补全字段必须写入 `Assumptions`
- 高风险字段必须进入待确认列表

### 4.4 规则引擎

#### 模块目标

负责约束、校验、判定可行性，并为参数化建模提供边界条件。

#### 需要什么

- 规则配置
- 规则执行器
- 规则版本管理
- 求解器

建议技术：

- `Python`
- `YAML/JSON`
- `OR-Tools CP-SAT`

#### 输入

- `DesignIntent`
- `ParsedDrawingModel`
- 项目规则集
- 地区规则集

#### 输出

- `RuleCheckResult`
- `RuleViolations`
- `RuleWarnings`
- `SolverConstraints`

#### 该怎么做

1. 先运行确定性规则校验。
2. 再运行布局和面积等求解类规则。
3. 按硬约束、软约束、偏好项分类处理。
4. 输出是否可继续建模，以及后续模块需要的约束条件。

#### 实施要点

- 规则必须带版本号
- 规则结果必须可解释
- 需支持地区规则覆盖项目默认规则

### 4.5 参数化建模 / 执行计划

#### 模块目标

根据规则结果和设计意图生成明确的建模执行路径。

#### 需要什么

- 编排器
- 参数化模板库
- 重生成策略
- 依赖关系管理

建议技术：

- `Python`
- `Pydantic`
- 自定义模板库

#### 输入

- `DesignIntent`
- `RuleCheckResult`
- `ParsedDrawingModel`
- 当前模型版本

#### 输出

- `ModelingPlan`
- `MassingOption`
- `FloorPlanResult`
- `AffectedModules`
- `RegenerationScope`

#### 该怎么做

1. 判断是首次生成还是增量修改。
2. 判断走 `cad_to_bim`、`text_to_massing` 还是 `element_batch_replace`。
3. 生成各子步骤执行顺序。
4. 标记哪些构件要创建、哪些要修改、哪些不变。
5. 输出可执行的建模计划。

#### 实施要点

- 必须支持局部重生成
- 执行计划必须可回放
- 每一步都要明确输入和输出对象

### 4.6 BIM 引擎

#### 模块目标

根据结构化建模指令，程序化生成 `BIM` 语义模型，并最终用于导出 `IFC`。

#### 需要什么

- `IFC` 建模库
- 几何生成能力
- 对象关系管理
- `GUID` 管理

建议技术：

- `IfcOpenShell`
- 可选 `OpenCASCADE`

#### 输入

- `ModelingPlan`
- `MassingOption`
- `FloorPlanResult`
- `FacadeResult`
- `ModelPatch`

#### 输出

- 内部 `BIM` 语义模型
- `IfcProject / IfcSite / IfcBuilding / IfcBuildingStorey`
- `IfcWall / IfcWindow / IfcDoor / IfcSlab / IfcSpace`
- 构件索引与版本映射

#### 该怎么做

1. 创建 `IfcProject / IfcSite / IfcBuilding / IfcBuildingStorey`。
2. 设置坐标系、原点、标高和对象放置关系。
3. 创建空间对象和主体构件。
4. 建立楼层归属、宿主关系、属性集关系。
5. 为对象生成参数化几何表达。
6. 执行构件替换、局部更新和属性变更。
7. 输出内部 `BIM` 模型交给自检模块。

#### 实施要点

- `GUID` 必须稳定
- 楼层、墙、窗、空间关系必须完整
- 内部主模型优先保留语义，不只保存三角网格

### 4.7 系统自检

#### 模块目标

在导出 `IFC` 前完成系统内部验证，确保模型可交付、可打开、可解释。

#### 需要什么

- `IFC` 读取和比对能力
- 规则复检器
- 几何校验器
- 需求落实检查器

建议技术：

- `IfcOpenShell`
- `IfcTester`
- `IfcDiff`
- 自定义校验器

#### 输入

- 当前 `BIM` 模型
- `DesignIntent`
- `ModelPatch`
- `RuleCheckResult`
- 上一个版本模型

#### 输出

- `ValidationReport`
- `ValidationStatus`
- `Issue[]`
- `AffectedElements`
- `FixSuggestion`

#### 该怎么做

1. 检查 `IFC` 结构是否完整。
2. 检查几何是否存在明显异常。
3. 检查构件与楼层、宿主、空间关系是否正确。
4. 检查规则是否满足。
5. 检查用户提出的修改是否真正落实。
6. 根据结果决定通过、警告或失败。

#### 实施要点

- 自检必须在导出前执行
- `fatal/error` 问题默认阻断导出
- 自检结果必须可追溯到具体构件

### 4.8 自检不通过回退

#### 模块目标

根据自检失败原因，将任务回退到正确的模块进行修复，而不是盲目全链路重跑。

#### 需要什么

- 问题分类器
- 回退策略
- 任务重试机制

建议技术：

- 任务编排器
- `Redis` 队列
- 自定义问题路由器

#### 输入

- `ValidationReport`
- 当前 `ModelingPlan`
- 模块状态信息

#### 输出

- `RetryTask`
- `RollbackTarget`
- `RepairInstruction`

#### 该怎么做

1. 根据问题类型定位来源模块。
2. 若是语义关系问题，回退 `BIM` 引擎。
3. 若是规则冲突问题，回退 `规则引擎` 或 `执行计划`。
4. 若是需求识别错误，回退 `AI 需求转化`。
5. 修复后重新触发建模与自检。

#### 实施要点

- 不通过不应默认全量重建
- 回退链路要有最大重试次数
- 每次回退都要记录原因和修复动作

### 4.9 输出 IFC 文件

#### 模块目标

将通过系统自检的内部 `BIM` 模型导出为可交付的 `IFC` 文件。

#### 需要什么

- `IFC` 导出器
- 文件版本管理
- 导出日志

建议技术：

- `IfcOpenShell`
- 对象存储
- 数据库版本表

#### 输入

- 自检通过的内部 `BIM` 模型
- 导出配置
- 项目版本信息

#### 输出

- `model.ifc`
- `validation.json`
- `intent.json`
- 可选 `preview.glb`

#### 该怎么做

1. 检查导出门禁是否通过。
2. 按项目要求生成正式 `IFC` 文件。
3. 保存导出日志和模型版本。
4. 将结果写入对象存储并登记到数据库。

#### 实施要点

- 未通过自检不得正式导出
- 文件命名必须带项目和版本信息
- 导出结果必须和自检报告绑定

### 4.10 Revit 查看与使用

#### 模块目标

供用户使用 `Revit` 打开 `IFC` 文件进行结果查看和后续使用。

#### 需要什么

- `Revit`
- `IFC` 文件
- 导出说明或自检报告

#### 输入

- `model.ifc`
- `validation.json`

#### 输出

- 用户查看结果
- 用户人工反馈

#### 该怎么做

1. 用户在 `Revit` 中打开或链接 `IFC` 文件。
2. 用户检查楼层、墙体、门窗、空间和尺寸是否符合预期。
3. 若发现问题，用户通过系统再次发起修改请求。

#### 实施要点

- `Revit` 不是系统内核模块
- `Revit` 不承担系统自动验证职责
- 用户反馈应重新进入需求转化或修改流程

## 5. 推荐技术栈汇总

- 输入接入：`React + FastAPI + PostgreSQL + MinIO/S3`
- 图纸解析：`DWG TrueView + ezdxf + PyMuPDF + Tesseract OCR`
- AI 转化：`GPT-5.4 + GPT-5.4-nano + Structured Outputs + Function Calling`
- 规则求解：`Python + YAML/JSON + OR-Tools`
- BIM 建模：`IfcOpenShell + OpenCASCADE`
- 系统自检：`IfcOpenShell + IfcTester + IfcDiff`
- 导出与交付：`IfcOpenShell + 对象存储`
- 最终查看：`Revit`

## 6. 结论

模块设计的核心原则如下：

- 输入、解析、需求转化、规则、建模、自检、导出必须职责分离
- `AI` 负责理解与转化，不直接负责几何建模
- `BIM` 引擎负责程序化创建 `IFC` 语义模型
- 系统必须先自检，再导出
- `Revit` 是最终查看与使用工具，而不是系统内部验证引擎
