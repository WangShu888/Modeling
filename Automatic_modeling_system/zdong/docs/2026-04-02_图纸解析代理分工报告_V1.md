# 图纸解析代理分工报告 V1

## 1. 任务背景

本报告用于把《建筑自动建模图纸解析模块修改方案 V1》落到当前代码仓库的可执行代理分工上，目标不是继续在 `app/drawing_parser.py` 里多人并行硬改，而是先按职责拆出新解析子模块，再由主代理统一接入现有 `ParsedDrawingModel -> pipeline` 链路。

当前工作区在登记时没有其他活动 agent，占用模块冲突为零，可以启动并行改造。

## 2. 当前代码约束

当前实现的共享热点只有 3 个：

- `app/drawing_parser.py`
  现有真实解析主入口，承担 DXF/DWG/PDF 解析、图元识别、楼层候选、片段生成和待审问题输出。
- `app/models.py`
  定义 `ParsedDrawingModel`、`DrawingEntityRecord`、`DrawingFragmentRecord`、`GridAxis`、`PendingReviewItem` 等输出契约。
- `app/pipeline.py`
  `ModelingPipeline` 通过 `RealDrawingParser()` 消费解析结果，后续 BIM、校核、导出都依赖这里的字段兼容。

结论：

- 多个代理不能同时编辑 `app/drawing_parser.py`、`app/models.py`、`app/pipeline.py`
- 共享契约和最终集成必须集中由主代理负责
- 子代理只应在新的独立模块文件内工作，避免写冲突

## 3. 并行改造原则

- 主代理只负责共享入口、共享模型、兼容接线和最终回归。
- 子代理只负责 `app/parser/` 下的新模块和对应新测试文件。
- 所有子代理都必须加测试，且不得改动其他代理负责的文件。
- `app/drawing_parser.py` 在本轮只作为编排入口，不再继续吸收新识别细节。
- 新模块先产出纯函数或小类接口，最后由主代理统一在 `DrawingParser.parse()` 中挂接。

## 4. 目标文件布局

建议新增 `app/parser/` 包，按方案中的职责拆成下列模块：

- `app/parser/dxf_input_adapter.py`
- `app/parser/view_classifier.py`
- `app/parser/floor_splitter.py`
- `app/parser/grid_recognizer.py`
- `app/parser/component_recognizer.py`
- `app/parser/annotation_binder.py`
- `app/parser/assembly_engine.py`
- `app/parser/validation_engine.py`
- `app/parser/compatibility_adapter.py`

为避免一次性改动过大，本轮先落 5 个并行工作面：

- 入口治理：`ViewClassifier` + `FloorSplitter`
- 轴网骨架：`GridRecognizer`
- 语义识别：`ComponentRecognizer` + `AnnotationBinder`
- 结果准入：`AssemblyEngine` + `ValidationEngine`
- 兼容适配：`ParserCompatibilityAdapter`

`DxfInputAdapter` 与共享模型字段扩展属于跨模块公共基础，暂由主代理统一负责。

## 5. 代理分工

### Agent-0 主代理 / 集成代理

职责：

- 负责共享契约设计和主链路接入
- 统一维护 `app/models.py`
- 统一维护 `app/drawing_parser.py`
- 统一维护 `app/pipeline.py`
- 创建 `app/parser/__init__.py` 和公共协议
- 合并各子代理模块到主解析流程
- 运行端到端回归并处理失败

文件所有权：

- `app/models.py`
- `app/drawing_parser.py`
- `app/pipeline.py`
- `app/parser/__init__.py`
- 端到端测试文件

不可并行共享的原因：

- 这 3 个文件是全链路耦合热点
- 任意子代理直接改这里都会引入合并冲突和行为漂移

### Agent-1 入口治理代理

职责：

- 实现 `ViewClassifier`
- 实现 `FloorSplitter`
- 建立主图/参考图分流规则
- 建立楼层区域切分结果

文件所有权：

- `app/parser/view_classifier.py`
- `app/parser/floor_splitter.py`
- `tests/test_view_classifier.py`
- `tests/test_floor_splitter.py`

阶段门槛：

- 能把“首层平面图 / 标准层平面图”标为主模型视图
- 能把“剖面图 / 详图 / 节点图”标为参考视图
- 能输出楼层级 `FloorRegion` 候选，不再把多楼层图元混成单包

### Agent-2 轴网骨架代理

职责：

- 实现 `GridRecognizer`
- 输出轴线、轴号、交点、间距和楼层轴网签名
- 为后续拼接提供稳定定位骨架

文件所有权：

- `app/parser/grid_recognizer.py`
- `tests/test_grid_recognizer.py`

阶段门槛：

- 能从现有 `TextAnnotationRecord` + 几何线段中识别主轴线
- 能形成楼层级 `GridNetwork` 表达
- 对打断轴线和偏轴至少给出保守合并或降级标记

### Agent-3 语义识别代理

职责：

- 实现 `ComponentRecognizer`
- 实现 `AnnotationBinder`
- 把几何候选、编号、尺寸、引线、块属性合并成主体构件结果

文件所有权：

- `app/parser/component_recognizer.py`
- `app/parser/annotation_binder.py`
- `tests/test_component_recognizer.py`
- `tests/test_annotation_binder.py`

阶段门槛：

- 墙、柱、门窗至少能形成“几何候选 + 标注确认”的双阶段结果
- 低置信度对象必须输出待确认，不得直接伪装成高置信度构件
- 输出字段必须可映射到现有 `DrawingEntityRecord.metadata`

### Agent-4 拼接与校核代理

职责：

- 实现 `AssemblyEngine`
- 实现 `ValidationEngine`
- 完成同层拼接、跨层拼接和问题隔离

文件所有权：

- `app/parser/assembly_engine.py`
- `app/parser/validation_engine.py`
- `tests/test_assembly_engine.py`
- `tests/test_validation_engine.py`

依赖：

- 依赖 Agent-1 的楼层切分结果
- 依赖 Agent-2 的轴网签名
- 依赖 Agent-3 的构件候选结果

阶段门槛：

- 能输出 `NeedReviewList` / `BlockedIssueList` 风格的结果
- 对楼层缺失、拼接重叠、轴网失配给出阻断或待审结论
- 不允许把拼接失败结果直接放行到建模链路

### Agent-5 兼容适配代理

职责：

- 实现 `ParserCompatibilityAdapter`
- 把新解析子模块结果映射回现有 `ParsedDrawingModel`
- 产出新旧字段差异说明和适配测试

文件所有权：

- `app/parser/compatibility_adapter.py`
- `tests/test_parser_compatibility_adapter.py`

边界说明：

- 不直接改 `app/pipeline.py`
- 不直接改 `app/models.py`
- 只产出适配器和适配测试，由主代理做最终接线

## 6. 执行顺序

### 第一批并行启动

- Agent-1 入口治理代理
- Agent-2 轴网骨架代理
- Agent-3 语义识别代理
- Agent-5 兼容适配代理

### 第二批启动

- Agent-4 拼接与校核代理

触发条件：

- Agent-1 已稳定输出楼层结果
- Agent-2 已稳定输出轴网签名
- Agent-3 已稳定输出构件候选和属性绑定结果

### 最终收口

- Agent-0 主代理统一接入 `DrawingParser.parse()`
- Agent-0 调整 `ParsedDrawingModel` 的兼容字段
- Agent-0 处理 `pipeline.py` 接线和全量回归

## 8. 当前轮次的实际派工结论

本轮不再把多个代理直接派到 `app/drawing_parser.py` 里竞争修改，而是采用以下落地策略：

- 主代理独占共享热点：`app/drawing_parser.py`、`app/models.py`、`app/pipeline.py`
- 子代理只在 `app/parser/` 新文件和各自测试文件中实现能力
- 先并行做“入口治理 / 轴网骨架 / 语义识别 / 兼容适配”
- 待上游数据契约稳定后，再启动“拼接与校核”

这样分工的原因很直接：

- 方案要求的是主链路重构，不是继续在单文件内堆条件分支
- 当前仓库最容易产生冲突的点是共享模型和主入口
- 先拆出新模块，后统一集成，才有可能让多个代理真实并行

## 9. 主代理待下发给子代理的任务模板

每个子代理下发任务时都必须包含以下约束：

- 你不是唯一在这个代码库工作的代理
- 只允许修改你负责的文件范围
- 不要回退其他代理的改动
- 必须补对应测试
- 如果发现需要改共享热点文件，只汇报，不直接修改

## 10. 交付目标

本分工完成后，图纸解析改造将从“多人共改一个大文件”转为“模块化并行开发 + 主代理集中集成”，与《建筑自动建模图纸解析模块修改方案 V1》的阶段目标保持一致，并能实际支撑后续代理协作落地。
