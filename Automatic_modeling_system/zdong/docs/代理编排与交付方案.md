# 标准版代理编排与交付方案

本方案严格对应 [模块分工建议.md](/workspace/自动建模系统/zdong/docs/模块分工建议.md) 的标准版分工，并以 [agent_plan.standard.json](/workspace/自动建模系统/zdong/app/config/agent_plan.standard.json) 作为机器可校验配置。

目标不是“每组各做一点”，而是让 7 个代理沿一条固定主链路形成可交付版本。

## 1. 标准版代理清单

| 顺序 | 代理 | 对应组 | 覆盖模块 | 核心职责 |
| --- | --- | --- | --- | --- |
| 1 | `agent_intake_asset` | 接入与资产组 | 4.1 | 接收图纸与文本，形成 `SourceBundle` |
| 2 | `agent_drawing_parser` | 图纸解析组 | 4.2 | 消费 `SourceBundle`，输出 `ParsedDrawingModel` |
| 3 | `agent_intent_ai` | AI 需求转化组 | 4.3 | 将需求与图纸解析结果转为 `DesignIntent` |
| 4 | `agent_rules_planner` | 规则与计划组 | 4.4, 4.5 | 生成 `RuleCheckResult` 和 `ModelingPlan` |
| 5 | `agent_bim_executor` | BIM 执行组 | 4.6 | 生成内部 BIM 语义模型 |
| 6 | `agent_validation_delivery` | 质量与回退组（兼导出交付） | 4.7, 4.8, 4.9 | 自检、精确回退、正式导出 |
| 7 | `agent_workbench_feedback` | 工作台与反馈组 | 4.10 | 版本展示、结果下载、反馈重新入链 |

## 2. 严格执行规则

- 每个代理只对自己负责的模块结果签字，不为其他组兜底。
- 每次交接必须交结构化对象，不接受“口头说明已完成”。
- `AI` 代理不得直接生成几何结果。
- `BIM` 代理不得直接消费自然语言。
- `Validation/Delivery` 代理在 `fatal/error` 未消除前必须阻断导出。
- `Workbench` 代理不得把 `Revit` 当作系统内部校验器。

## 3. 图纸解析代理的特殊要求

图纸解析组已经有现成的部分成果，因此 `agent_drawing_parser` 必须采用“承接既有成果”的方式执行，不能推倒重来。

已知应直接承接的落点：

- [drawing_parser.py](/workspace/自动建模系统/zdong/app/drawing_parser.py)
- [windows_dwg_converter_service.py](/workspace/自动建模系统/zdong/app/windows_dwg_converter_service.py)
- [test_dwg_conversion.py](/workspace/自动建模系统/zdong/tests/test_dwg_conversion.py)
- [test_remote_dwg_conversion.py](/workspace/自动建模系统/zdong/tests/test_remote_dwg_conversion.py)
- [test_windows_dwg_converter_service.py](/workspace/自动建模系统/zdong/tests/test_windows_dwg_converter_service.py)

执行要求：

- 先复用现有 `DWG -> DXF` 转换能力和测试资产。
- 再补齐 `LayerMap / GridMap / PendingReview` 等中间结果的一致性。
- 不得为了追求“从零统一风格”而重写已可用解析链路。

## 4. 主链路执行顺序

主链路固定如下：

`SourceBundle -> ParsedDrawingModel -> DesignIntent -> ModelingPlan -> 内部 BIM 语义模型 -> ValidationReport/ExportBundle -> 用户反馈`

各代理的启动顺序必须是：

1. `agent_intake_asset`
2. `agent_drawing_parser`
3. `agent_intent_ai`
4. `agent_rules_planner`
5. `agent_bim_executor`
6. `agent_validation_delivery`
7. `agent_workbench_feedback`

## 5. 代理交接对象

| 上游代理 | 下游代理 | 唯一正式交接对象 |
| --- | --- | --- |
| `agent_intake_asset` | `agent_drawing_parser` | `SourceBundle` |
| `agent_drawing_parser` | `agent_intent_ai` | `ParsedDrawingModel` |
| `agent_intent_ai` | `agent_rules_planner` | `DesignIntent` |
| `agent_rules_planner` | `agent_bim_executor` | `ModelingPlan` |
| `agent_bim_executor` | `agent_validation_delivery` | 内部 `BIM` 语义模型 |
| `agent_validation_delivery` | `agent_workbench_feedback` | `ExportBundle` |

任何代理如果绕开交接对象直接读下游文件，视为越权。

## 6. 版本交付门禁

要形成真正的版本交付，必须满足以下门禁：

1. `agent_intake_asset` 已产出可追溯的 `SourceBundle`
2. `agent_drawing_parser` 已输出可消费的 `ParsedDrawingModel`
3. `agent_intent_ai` 已输出严格 schema 化的 `DesignIntent`
4. `agent_rules_planner` 已给出带版本的规则结果和可回放 `ModelingPlan`
5. `agent_bim_executor` 已生成可检查的内部 BIM 语义模型
6. `agent_validation_delivery` 已完成自检，且仅在通过后导出 `model.ifc`
7. `agent_workbench_feedback` 已提供查看、下载和反馈闭环

缺任一项，都不算版本交付完成。

## 7. 推荐启动方式

推荐按两阶段推进：

### 阶段 A：主链路打通

- `agent_intake_asset`
- `agent_drawing_parser`
- `agent_intent_ai`
- `agent_rules_planner`
- `agent_bim_executor`
- `agent_validation_delivery`

目标：

- 先形成一个能从输入走到 `model.ifc` 的稳定主链路

### 阶段 B：交付闭环补齐

- `agent_workbench_feedback`

目标：

- 让交付结果可被查看、下载、反馈并重新进入下一个版本请求

## 8. 每个代理的完成标准

### `agent_intake_asset`

- 项目、请求、资产、版本 ID 完整
- 上传资产和文字需求绑定到同一个 `request_id`
- 输出标准 `SourceBundle`

### `agent_drawing_parser`

- 在现有成果基础上稳定输出 `ParsedDrawingModel`
- 对无法识别内容输出 `PendingReview`
- 不将业务判断混入解析结果

### `agent_intent_ai`

- 输出 `DesignIntent`
- 输出 `Assumptions / MissingFields / CompletionTrace`
- 高风险字段不隐式吞掉

### `agent_rules_planner`

- 先规则后计划
- 输出 `RuleCheckResult` 和 `ModelingPlan`
- 明确局部重生成范围

### `agent_bim_executor`

- 以语义对象为核心建模
- 楼层、宿主、空间关系完整
- 不绕过验证门禁

### `agent_validation_delivery`

- 自检失败精确回退
- 自检通过后生成 `model.ifc`
- 导出结果绑定 `validation.json` 与 `intent.json`

### `agent_workbench_feedback`

- 展示版本与交付产物
- 提供问题反馈入口
- 将反馈重新转成新的请求链路

## 9. 一句话执行口径

标准版执行时，7 个代理不是并列做功能，而是按固定顺序接棒，最终以 `agent_validation_delivery` 产出正式交付物、以 `agent_workbench_feedback` 形成反馈闭环。
