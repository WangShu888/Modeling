# 建筑自动建模模块分工建议

基于 [模块要求.md](/workspace/自动建模系统/zdong/docs/模块要求.md) 的主链路：

`图纸、文字信息 -> CAD/PDF解析 -> AI需求转化 -> 规则引擎 -> 参数化建模/执行计划 -> BIM引擎 -> 系统自检 -> 回退修复 -> IFC导出 -> Revit查看`

本文档给出的不是纯功能清单，而是可直接用于团队排人的研发分工方案。

## 1. 分工原则

- 一组只对一层核心能力负责，不跨层兜底。
- `AI` 组负责“理解需求”，不直接负责几何生成。
- `BIM` 组负责“生成语义模型”，不负责解释自然语言需求。
- 自检与回退必须独立于建模执行，避免自己给自己放行。
- 平台编排要单独有人负责，否则容易出现模块都能跑、主链路不稳定的问题。

## 2. 推荐分工方式

建议按 8 个责任域分工，而不是机械地按 10 个业务模块一人一个模块拆。

| 责任域 | 对应模块 | 主责 | 不负责 | 当前代码落点 |
| --- | --- | --- | --- | --- |
| 1. 接入与资产组 | 4.1 图纸、文字信息接入 | 上传、项目表单、资产存储、版本元数据、`SourceBundle` 组装 | 图纸识别、规则判断、BIM 建模 | `app/main.py` `app/assets.py` `app/store.py` |
| 2. 图纸解析组 | 4.2 CAD/PDF 解析 | `DWG -> DXF`、`DXF/PDF` 解析、单位归一化、轴网/尺寸/文字/楼层候选识别 | 用户意图理解、法规判断、IFC 导出 | `app/drawing_parser.py` |
| 3. AI 需求转化组 | 4.3 AI 需求转化 | `DesignIntent`、`MissingFields`、`Assumptions`、`ModelPatch`、结构化输出约束 | 规则求解、几何建模、导出验证 | `app/intent_service.py` `app/config/intent_defaults.json` |
| 4. 规则与计划组 | 4.4 规则引擎 + 4.5 参数化建模/执行计划 | 规则集、求解约束、建模策略判断、`ModelingPlan`、局部重生成范围 | IFC 几何实现、最终导出 | `app/planning.py` `app/config/rules.default.json` `app/config/planner.default.json` |
| 5. BIM 执行组 | 4.6 BIM 引擎 | `IfcProject` 到构件级对象生成、楼层关系、宿主关系、稳定 `GUID`、模型增量修改 | 需求解释、规则裁决、验收放行 | `app/ifc_runtime.py` 以及后续新增 BIM 语义生成模块 |
| 6. 质量与回退组 | 4.7 系统自检 + 4.8 自检不通过回退 | 结构校验、几何校验、需求落实检查、问题分类、回退路由、重试策略 | 直接修改建模逻辑、前端交互 | 当前主要在 `app/pipeline.py`，建议后续独立为 validation/retry 模块 |
| 7. 导出交付组 | 4.9 输出 IFC 文件 | 导出门禁、`IFC`/`validation.json`/`intent.json` 产物落盘、版本登记、对象存储发布 | 解析识别、规则判断 | 当前主要在 `app/pipeline.py` 和导出目录逻辑，建议后续独立 export 模块 |
| 8. 工作台与反馈组 | 4.10 Revit 查看与使用 + 系统前端 | Web 工作台、项目流转可视化、问题反馈入口、Revit 使用说明和结果闭环 | 内部自动验证、IFC 语义建模 | `web/*`，以及 API 对接层 |

## 3. 每组的交付边界

### 3.1 接入与资产组

上游面对用户，下游只对解析组输出标准化 `SourceBundle`。

必须交付：

- 上传接口
- 项目和请求 ID 体系
- 文件元数据和版本关系
- 原始文件可追溯能力

边界要求：

- 不能把“解析成功”当成自己的完成条件
- 只保证文件和表单进入系统且可被后续消费

### 3.2 图纸解析组

这是整个系统的“事实提取层”，对图纸内容负责。

必须交付：

- `ParsedDrawingModel`
- `LayerMap`
- `GridMap`
- `DimensionEntities`
- `TextAnnotations`
- `StoreyCandidates`
- 待确认清单

边界要求：

- 不能根据用户自然语言擅自修改识别结果
- 不能替规则组做可行性判断

### 3.3 AI 需求转化组

这是“设计意图解释层”，对结构化意图负责。

必须交付：

- 严格遵循 schema 的 `DesignIntent`
- `Assumptions`
- `MissingFields`
- 构件级补丁对象
- 字段来源追踪

边界要求：

- 不直接产出 BIM 几何
- 不把自由文本直接传给后端作为最终指令

### 3.4 规则与计划组

这是“能不能做、怎么做”的决策层。

必须交付：

- `RuleCheckResult`
- `RuleViolations / RuleWarnings`
- `SolverConstraints`
- `ModelingPlan`
- `AffectedModules`
- `RegenerationScope`

边界要求：

- 不负责落地 IFC 构件
- 不负责兜底缺失字段的业务解释

### 3.5 BIM 执行组

这是“把计划变成模型”的执行层。

必须交付：

- 内部 BIM 语义模型
- `IfcProject / IfcSite / IfcBuilding / IfcBuildingStorey`
- 墙、门、窗、板、空间等主体构件
- 构件索引与版本映射

边界要求：

- 只接受结构化输入，不消费自然语言
- 不能自己跳过规则或自检直接导出

### 3.6 质量与回退组

这是“系统守门员”。

必须交付：

- `ValidationReport`
- `Issue[]`
- `RollbackTarget`
- `RepairInstruction`
- 回退重试记录

边界要求：

- 自检规则要独立于 BIM 组
- 失败要精确回退，不允许默认全量重跑

### 3.7 导出交付组

这是“正式产物交付层”。

必须交付：

- 正式 `model.ifc`
- `validation.json`
- `intent.json`
- 导出日志和版本登记

边界要求：

- 未通过自检不得放行
- 导出版本必须可追溯到请求、规则版本和意图版本

### 3.8 工作台与反馈组

这是“用户操作和反馈闭环层”。

必须交付：

- 项目工作台
- 任务状态展示
- 版本查看
- 导出结果下载
- 问题反馈重新入链能力

边界要求：

- `Revit` 仅是用户查看工具，不是系统内部验证器
- 用户反馈进入新请求，不直接改底层模型文件

## 4. 建议负责人画像

| 责任域 | 建议负责人画像 |
| --- | --- |
| 接入与资产组 | 熟悉 `FastAPI`、文件存储、数据库、接口建模的后端工程师 |
| 图纸解析组 | 熟悉 `CAD/DXF/PDF`、几何数据处理、工程图识别的算法/后端工程师 |
| AI 需求转化组 | 熟悉提示词、`JSON Schema`、结构化输出和业务字段建模的 AI 工程师 |
| 规则与计划组 | 熟悉约束建模、规则系统、编排器和状态机的后端工程师 |
| BIM 执行组 | 熟悉 `IfcOpenShell`、BIM 语义、对象关系和几何表达的建模工程师 |
| 质量与回退组 | 熟悉测试体系、校验器、错误路由和流程编排的后端/QA 工程师 |
| 导出交付组 | 熟悉导出链路、版本管理、对象存储和交付规范的后端工程师 |
| 工作台与反馈组 | 熟悉 `React`、流程可视化、表单和文件交互的前端工程师 |

## 5. 推荐的人力安排

### 5.1 标准版：6 到 8 人

推荐这样排：

- 1 人负责接入与资产组
- 1 到 2 人负责图纸解析组
- 1 人负责 AI 需求转化组
- 1 人负责规则与计划组
- 1 到 2 人负责 BIM 执行组
- 1 人负责质量与回退组，可兼导出交付
- 1 人负责工作台与反馈组

适用场景：

- 你希望并行开发
- 后续会做增量修改、局部重建和版本管理

### 5.2 精简版：4 人

如果团队人少，建议合并为 4 条线：

- A 线：接入与工作台
  - 负责 4.1、4.10
- B 线：图纸解析
  - 负责 4.2
- C 线：AI + 规则 + 执行计划
  - 负责 4.3、4.4、4.5
- D 线：BIM + 自检 + 回退 + 导出
  - 负责 4.6、4.7、4.8、4.9

注意：

- 精简版能启动，但后期最容易堵在 C 线和 D 线
- 一旦进入联调阶段，应优先把自检从 BIM 负责人手里拆出来

## 6. 建议的协作接口

为避免反复扯皮，建议按对象交接，而不是按口头说明交接：

- 接入组 -> 解析组：`SourceBundle`
- 解析组 -> AI 组：`ParsedDrawingModel`
- AI 组 -> 规则组：`DesignIntent`
- 规则组 -> 计划组：`RuleCheckResult`
- 计划组 -> BIM 组：`ModelingPlan`
- BIM 组 -> 质量组：内部 `BIM` 语义模型
- 质量组 -> 回退/导出组：`ValidationReport`
- 导出组 -> 工作台组：`ExportBundle`

## 7. 当前仓库建议的文件归属

按现有代码结构，建议先这样认领：

- 平台与 API：`app/main.py` `app/pipeline.py`
- 数据模型：`app/models.py`
- 接入与资产：`app/assets.py` `app/store.py`
- 图纸解析：`app/drawing_parser.py`
- AI 需求转化：`app/intent_service.py`
- 规则与计划：`app/planning.py`
- BIM 运行时：`app/ifc_runtime.py`
- 配置资产：`app/config/*.json`
- 前端工作台：`web/*`

其中 `app/pipeline.py` 当前承担了过多编排职责，后续建议逐步拆为：

- `pipeline/orchestrator`
- `validation/service`
- `retry/router`
- `export/service`

## 8. 开发顺序建议

推荐顺序如下：

1. 先打通 4.1 -> 4.5，保证请求可以稳定形成 `ModelingPlan`
2. 再完成 4.6，先生成最小可用 `IFC` 语义骨架
3. 然后补 4.7 和 4.9，建立“先自检再导出”的门禁
4. 最后补强 4.8 和 4.10，形成问题闭环和版本迭代能力

如果一开始就同时铺开全部模块，最容易出现的问题是：

- 各模块都有 demo
- 但主链路没有一个版本能完整交付

## 9. 一句话结论

如果你要实际排人，最合理的拆法是：

- 前台入口一组
- 图纸解析一组
- AI 意图一组
- 规则与计划一组
- BIM 执行一组
- 质量回退与导出一组
- 工作台反馈一组

其中最核心、最不能混在一起的是三条线：

- `AI 需求转化`
- `规则/执行计划`
- `BIM 语义建模`

这三者一旦由同一人或同一模块混做，后面联调和排错成本会明显升高。
