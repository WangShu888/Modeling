# 图纸解析流程文档

## 1. 文档范围

本文档描述当前 `zdong` 项目里“图纸解析”这一段的实际实现流程，重点对应以下模块：

- `zdong/app/pipeline.py`
- `zdong/app/drawing_parser.py`
- `zdong/app/storey_inference.py`
- `zdong/app/models.py`

这里的“图纸解析”不是单指 CAD 文件读取，而是指从请求输入、图纸识别、楼层归并，到把解析结果提供给后续 BIM 建模的完整上游链路。

## 2. 主流程概览

当前主链路分成两种调用方式：

1. 仅解析图纸
   - `ModelingPipeline.parse_request()`
   - 适合只看解析结果，不继续建模的场景。
2. 解析后继续建模
   - `ModelingPipeline._run_pipeline()`
   - 顺序执行 `drawing_parser -> intent_transformer -> rule_engine -> planner -> bim_engine -> validation -> export`

图纸解析在主链路中的位置如下：

```text
请求/项目/附件
  -> SourceBundleBuilder.build()
  -> DrawingParser.parse()
     -> _parse_asset()
        -> _parse_dxf() / _parse_dwg() / _parse_pdf()
     -> 聚合 ParsedDrawingModel
  -> storey_inference 推断楼层与视图角色
  -> BimEngine._source_layouts() 消费解析结果
  -> 后续建模/校验/导出
```

## 3. 输入组装阶段

### 3.1 SourceBundleBuilder 组装输入

入口在 `pipeline.py` 的 `SourceBundleBuilder.build()`。

这一阶段会把项目、请求、附件整理为统一的 `SourceBundle`，主要包含：

- `project_id`
- `request_id`
- `version_id`
- `prompt`
- `source_mode_hint`
- `assets`
- `form_fields`

其中：

- `assets` 是本次请求关联的图纸附件列表，元素类型为 `AssetRecord`
- `form_fields` 由请求中的结构化字段和 `metadata` 合并而来，后续会被解析器用于读取 `unit`、`north_angle` 等先验信息

## 4. DrawingParser 总控流程

入口在 `drawing_parser.py` 的 `DrawingParser.parse(bundle)`。

这一层是总控聚合器，不直接关心某一种文件的细节，而是负责：

1. 初始化全局聚合容器
   - `recognized_layers`
   - `grid_map`
   - `dimension_details`
   - `text_items`
   - `detected_entities`
   - `storey_candidate_details`
   - `storey_elevations`
   - `pdf_assets`
   - `pending_review`
   - `unresolved_entities`
   - `entity_summary`
2. 从 `bundle.form_fields` 读取初始先验
   - 默认单位 `unit`，缺省值为 `mm`
   - 默认北向角 `north_angle`，缺省值为 `0.0`
3. 遍历每个附件，调用 `_parse_asset(asset)`
4. 合并各附件解析结果
5. 处理跨附件冲突
   - 单位冲突会写入 `pending_review`
   - 原点通过 `_resolve_coordinate_reference()` 合并
   - 北向通过 `_resolve_north_angle()` 合并
6. 汇总生成 `ParsedDrawingModel`

如果请求没有上传任何图纸，解析器不会抛异常，而是把 `"No drawing assets were uploaded."` 写入 `unresolved_entities`。

## 5. 单附件分发流程

入口在 `DrawingParser._parse_asset(asset)`。

### 5.1 预处理

在真正读取文件前，解析器先做两件事：

1. 生成 `descriptor`
   - 由 `filename + description` 组成
2. 从附件名和描述中生成回退楼层候选
   - 通过 `_descriptor_storey_candidates()`
   - 可产生 `standard_floor`、`facade_reference`、`section_reference`

### 5.2 路径解析

通过 `_resolve_asset_path()` 在以下候选路径中找真实文件：

- `asset.path`
- `asset.filename`
- `workspace_root / asset.filename`
- `workspace_root / asset.path`（当 `asset.path` 是相对路径时）

如果文件不存在，则输出：

- `pending_review.category = asset_missing`
- `unresolved_entities` 中记录缺失原因

### 5.3 按文件格式分支

目前支持三类输入：

- `.dxf` -> `_parse_dxf()`
- `.dwg` -> `_parse_dwg()`
- `.pdf` -> `_parse_pdf()`

其他扩展名直接归类为 `unknown`，并写入 `unresolved_entities`。

## 6. DXF 解析流程

### 6.1 文件读取

`_parse_dxf()` 使用 `ezdxf.readfile(path)` 打开图纸。

如果读取失败：

- `pending_review.category = dxf_parse_failed`
- 同时写入 `unresolved_entities`

如果读取成功，进入 `_parse_dxf_document()`。

### 6.2 DXF 实体遍历

`_parse_dxf_document()` 遍历 `doc.modelspace()`，按实体类型处理。

同时会先基于图层名调用 `_guess_semantic_role(layer_name)` 做语义猜测，例如：

- `wall`
- `grid`
- `room`
- `space`
- `door`
- `window`
- `site_boundary`
- `facade`

### 6.3 线类实体处理

实体类型：`LINE`、`XLINE`、`RAY`

处理动作：

- 统计 `entity_summary["lines"]`
- 计算起止点、包围盒、方向
- 若图层语义是 `grid`，写入 `grid_map`
- 若图层语义是 `wall/site_boundary/facade`，写入 `detected_entities`

典型输出类别：

- `wall_line`
- `site_boundary_line`
- `facade_line`

### 6.4 多段线处理

实体类型：`LWPOLYLINE`、`POLYLINE`

处理动作：

- 统计 `entity_summary["polylines"]`
- 提取点集
- 判断是否闭合

识别规则：

- 闭合且图层语义为 `room/space` -> `room_boundary`
- 闭合且图层语义为 `site_boundary` -> `site_boundary`
- 图层语义为 `wall` -> `wall_path`
- 其他闭合图形 -> 暂存为 `boundary_candidate`

补充逻辑：

- 如果整张图没有显式识别到 `site_boundary`，但存在最大的闭合未知边界，则会把它作为 `site_boundary_candidate`
- 同时追加一条 `pending_review.category = site_boundary_inferred`

### 6.5 块参照处理

实体类型：`INSERT`

处理动作：

- 统计 `entity_summary["blocks"]`
- 结合块名和图层名再次猜测语义
- 识别门窗/轴网标记

典型输出类别：

- `door_block`
- `window_block`
- `grid_marker`

### 6.6 文本处理

实体类型：`TEXT`、`MTEXT`、`ATTRIB`、`ATTDEF`

处理动作：

- 统计 `entity_summary["texts"]`
- 抽取文本内容
- 生成 `TextAnnotationRecord`
- 用 `_classify_text_semantics()` 标记语义
- 用 `_extract_grid_label()` 抽轴号
- 用 `_extract_dimension_records_from_text()` 抽尺寸
- 用 `_extract_elevations()` 抽标高
- 用 `_extract_north_angle()` 抽北向角
- 如果语义是 `view_marker`，调用 `_append_view_marker_candidates()` 生成视图/楼层候选

这一阶段是当前楼层判断的重要来源之一。

例如：

- `二层平面图` 会被识别为 `standard_floor`，并保留原始文本候选
- `1-1剖面图` 会生成 `section_reference`
- `南立面图` 会生成 `facade_reference`

其中 `_append_view_marker_candidates()` 已对 callout 类标记做过滤，避免仅由符号前缀组成的详图索引误导整张图的主视图判断。

### 6.7 标注实体处理

实体类型：名称中包含 `DIMENSION`

处理动作：

- 统计 `entity_summary["dimensions"]`
- 尝试读取尺寸值
- 生成 `DimensionEntityRecord`

### 6.8 DXF 输出结果

DXF 分支最终返回 `_AssetParseResult`，核心内容包括：

- `units`
- `origin`
- `north_angle`
- `recognized_layers`
- `layer_map`
- `grid_map`
- `dimension_details`
- `text_items`
- `detected_entities`
- `storey_candidates`
- `storey_elevations_m`
- `pending_review`

## 7. DWG 解析流程

DWG 分支本质上是“先转 DXF，再走 DXF 解析”。

### 7.1 版本识别

`_parse_dwg()` 先调用 `_read_dwg_version()` 读取文件头，用于：

- 记录 `DWG version ...` 文本注释
- 映射 ODA 转换目标版本

### 7.2 DWG 转 DXF

`_convert_dwg_to_dxf()` 支持两种路径：

1. 远程 Windows 转换服务
   - 由 `JIANMO_DWG_CONVERTER_URL` 或 `JIANMO_WINDOWS_CONVERTER_URL` 配置
   - 调用 `_convert_dwg_to_dxf_via_remote_service()`
2. 本地 ODA File Converter
   - 由 `JIANMO_ODAFC_PATH` 或 `ODA_FILE_CONVERTER_PATH` 指向
   - Linux 下如果是 `.exe`，会尝试通过 `wine` 执行
   - 若有 `xvfb-run`，会自动加上虚拟显示前缀

失败路径会转成结构化问题，而不是直接中断主流程：

- `dwg_converter_missing`
- `dwg_conversion_failed`
- `dwg_converted_dxf_invalid`

### 7.3 转换后复用 DXF 解析

转换成功后：

1. 用 `ezdxf.readfile(converted_path)` 读取临时 DXF
2. 调用 `_parse_dxf_document()`
3. 附加一条 `dwg_version` 文本注释
4. 删除临时转换目录

## 8. PDF 解析流程

### 8.1 文件读取

`_parse_pdf()` 使用 `fitz.open(path)` 打开 PDF。

如果失败：

- `pending_review.category = pdf_parse_failed`
- 写入 `unresolved_entities`

### 8.2 页级判定

遍历每一页时，分别统计：

- 文本块数量
- 向量图元数量
- 图片数量

再通过 `_classify_pdf_page()` 判断页面类型：

- `vector`
- `scanned`
- `hybrid`
- 其他组合

### 8.3 文本提取

对 `page.get_text("blocks")` 返回的文本块逐一处理，逻辑与 CAD 文本类似：

- 生成 `TextAnnotationRecord`
- 抽取轴号、尺寸、标高、北向角
- 命中视图标题时写入 `storey_candidates`

### 8.4 向量提取

对 `page.get_drawings()` 返回的向量项逐一处理。

当前重点识别：

- 线段 -> `wall_line` / `facade_line` / `vector_line`
- 矩形或轮廓 -> `room_boundary` / `site_boundary` / `block_region` / `vector_outline`

### 8.5 OCR 补充

如果页类型中存在 `scanned`：

- 当前环境支持 OCR，则调用 `_run_optional_pdf_ocr()`
- 不支持 OCR，则写入 `pending_review.category = scanned_pdf_requires_ocr`

此外，如果整份 PDF 最终被判定为 `scanned`，`_finalize_asset_quality()` 还会追加：

- `pending_review.category = pdf_scan_review`

### 8.6 PDF 输出结果

PDF 分支除常规解析结果外，还会输出 `PdfAssetInfo`，记录：

- 页数
- 文档类型
- 各类型页面数量
- 是否尝试 OCR
- 当前环境是否支持 OCR

## 9. 质量补充与告警

每个附件解析完成后，`_finalize_asset_quality()` 会补充质量告警：

- CAD 图未识别出墙体 -> `wall_detection_low_confidence`
- CAD 图未识别出轴网 -> `grid_not_detected`
- PDF 为扫描件 -> `pdf_scan_review`

这些问题不会阻断解析，但会进入 `ParsedDrawingModel.pending_review`，供后续规则检查和人工确认使用。

## 10. 跨附件聚合输出

`DrawingParser.parse()` 在遍历完所有附件后，会把结果汇总成 `ParsedDrawingModel`。

核心输出字段如下：

- `asset_kinds`
- `units`
- `origin`
- `recognized_layers`
- `layer_map`
- `grid_map`
- `dimension_details`
- `text_annotations`
- `text_annotation_items`
- `detected_entities`
- `storey_candidates`
- `storey_candidate_details`
- `storey_elevations_m`
- `north_angle`
- `pdf_assets`
- `pdf_modes_detected`
- `site_boundary_detected`
- `space_boundaries_detected`
- `pending_review`
- `unresolved_entities`
- `entity_summary`

这一步完成后，图纸解析阶段本身就结束了。

## 11. 楼层推断流程

图纸解析的结果并不会直接等于“楼层结果”。楼层归并在 `storey_inference.py` 中完成。

### 11.1 楼层文本识别

`infer_storey_key(text)` 负责把文本归一为标准楼层键值，当前支持：

- `1F`、`2F`
- `B1`、`B2`
- `RF`
- 中文 `一层`、`二层`
- 中文 `地下一层`、`地下二层`
- 中文 `屋面`

### 11.2 视图角色识别

`infer_asset_view_role(asset_name, candidate_names)` 用于判断一张图更像：

- `plan`
- `section`
- `facade`
- `unknown`

判断依据来自两部分：

- 附件名
- `storey_candidate_details` 中的候选文本

设计目标是让“平面图”优先级高于零散的剖面/立面引用，避免详图索引把整张图误判。

### 11.3 按资产推断楼层

`infer_parsed_asset_storeys(parsed)` 会：

1. 收集每个附件的候选文本
2. 只挑选与建模相关的解析实体来源
   - `wall_line`
   - `wall_path`
   - `door_block`
   - `window_block`
   - `room_boundary`
3. 先尝试从附件名取楼层
4. 取不到再从候选文本里找楼层
5. 若仍没有显式楼层，则在合理范围内回退到默认楼层

最终输出：

- `asset_name -> storey_key`

## 12. 解析结果如何被下游消费

解析模块的直接消费者是 `pipeline.py` 中的 `BimEngine`。

### 12.1 形成可建模楼层布局

`BimEngine._source_layouts(parsed_drawing)` 会：

1. 读取 `storey_candidate_details`
2. 调用 `infer_parsed_asset_storeys(parsed_drawing)` 生成楼层映射
3. 读取 `parsed_drawing.detected_entities`
4. 过滤掉非建模核心类别，仅保留：
   - `wall_line`
   - `wall_path`
   - `door_block`
   - `window_block`
   - `room_boundary`
5. 再过滤掉 `section/facade` 视图
6. 按 `storey_key` 对实体归组
7. 计算每层的原点、宽度、进深、比例尺和实体集合

输出结果是若干 `layout`，每个 `layout` 对应一层可直接建模的平面布局。

### 12.2 驱动 BIM 建模

`BimEngine.build()` 会优先消费这些 `source_layouts`：

- 如果解析到了有效楼层布局，则按图纸几何生成空间、墙、门、窗等 BIM 元素
- 如果没有解析到可用布局，则退回模板建模

因此，图纸解析阶段的质量，直接决定后续是：

- `geometry_source = parsed_drawing`
- 还是 `geometry_source = template_fallback`

## 13. 当前流程的关键特点

当前实现有几个明显特征：

1. DWG 不直接解析，而是统一先转成 DXF
2. DXF 是当前最完整的解析分支
3. PDF 能解析文本和部分向量，但可靠性依赖页面类型与 OCR 条件
4. 楼层判断不是只看文件名，而是综合文件名、图内文本、实体类别共同判断
5. 解析阶段大量使用 `pending_review` 和 `unresolved_entities` 保留不确定性，而不是简单成功/失败二分
6. 解析结果已经不是“摘要文本”，而是后续 BIM 建模可直接消费的结构化中间模型

## 14. 一句话总结

当前图纸解析流程本质上是：

“先把请求和附件整理成统一输入，再按 `DXF / DWG / PDF` 分支提取图层、几何、文本、楼层候选和质量告警，汇总为 `ParsedDrawingModel`，然后用楼层推断逻辑筛出可建模的平面楼层布局，交给后续 BIM 引擎继续生成模型。”
