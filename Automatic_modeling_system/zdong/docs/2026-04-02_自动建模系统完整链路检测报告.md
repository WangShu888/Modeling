## 3. 发现的问题

### 3.1 测试基线问题

- `tests/test_dxf_delivery_pipeline.py` 原先写死旧样本目录，导致真实 DXF 回归失败
- 现已改为动态解析样本路径，问题已修复

### 3.2 构件识别风险

- 真实样本解析后 `detected_entities = 200`，正好触达当前上限
- 说明现有结果可能不是“真实识别总量”，而是“截断后的前 200 项”
- 当前系统已增加 `entity_detection_truncated` 告警

### 3.3 楼层关联拼接风险

- 同一张图中存在 `1F / 2F / RF` 显式候选
- 但 `source_storey_keys` 最终仅为 `['1F']`
- 根因是当前 `_source_layouts()` 仍按 `asset_name -> storey_key` 的单值映射组织几何，缺少单资产内按视图区块切分楼层的能力

## 4. 结论

- 当前版本已经具备可复现的完整主链路能力
- 当前版本还不具备可靠的“单图多楼层自动切分并拼接”能力
- 当前版本的构件识别结果可用于交付草案，但不能把 `detected_entities = 200` 直接当作识别上限真实值

## 5. 建议

1. 优先为单资产多楼层场景实现“按视图标题/边框/几何聚类分段”的楼层切分，再谈跨楼层拼接。
2. 将 `detected_entities`、`text_items`、`dimension_details` 的上限命中情况统一写入报告元数据，避免误判识别覆盖率。
3. 为真实 DXF 样本补一条长期稳定的回归入口，避免后续再次出现样本路径漂移。
