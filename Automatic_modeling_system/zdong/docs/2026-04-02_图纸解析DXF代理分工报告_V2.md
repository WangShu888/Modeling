# 图纸解析 DXF 代理分工报告 V2

## 1. 范围收敛

本轮只处理 `DXF` 解析，不处理以下范围：

- `PDF` 解析
- `DWG` 转 `DXF`
- OCR
- 远端转换服务

因此，代理分工不再为 `pdf_reader`、`conversion`、远端服务接入预留写权限，所有工作都围绕现有 `DXF -> ParsedDrawingModel -> pipeline` 主链路展开。

## 2. 当前仓库里的共享热点

当前必须由主代理独占的共享热点只有 3 个：

- `app/drawing_parser.py`
  现有 `DrawingParser.parse()`、`_AssetParseResult`、`_parse_asset()`、`_parse_dxf()` 和最终 `ParsedDrawingModel` 组装都在这里。
- `app/models.py`
  定义了解析结果契约，包括 `ParsedDrawingModel`、`DrawingEntityRecord`、`DrawingFragmentRecord`、`GridAxis`、`PendingReviewItem`。
- `app/pipeline.py`
  `ModelingPipeline` 通过 `RealDrawingParser()` 和 `ParsedDrawingModel` 直接消费解析结果。

结论：

- 子代理不得直接修改上述 3 个文件
- 所有新模块必须先在独立文件中实现，再由主代理统一接入

## 4. DXF-only 的实际拆分切口

基于现有 `drawing_parser.py` 的职责边界，本轮最适合并行拆出的新模块如下：

- `app/parser/common.py`
  承接文本归一化、几何基础工具、坐标和包围盒辅助函数
- `app/parser/dxf_reader.py`
  承接 `_parse_dxf_document()` 及 DXF 文档内图元扫描
- `app/parser/view_storey.py`
  承接视图语义识别、楼层候选和视图标题标记
- `app/parser/fragments.py`
  承接片段聚类、实体归片段、楼层片段构建和解析期诊断
- `app/parser/grid_recognizer.py`
  承接轴网识别和楼层轴网签名
- `app/parser/component_recognizer.py`
  承接墙柱门窗等构件候选提取
- `app/parser/annotation_binder.py`
  承接编号、尺寸、引线和块属性绑定
- `app/parser/assembly_engine.py`
  承接同层拼接和跨层拼接
- `app/parser/validation_engine.py`
  承接问题隔离和准入校核
- `app/parser/compatibility_adapter.py`
  把新解析模块结果适配回现有 `ParsedDrawingModel`

## 8. 本轮实际派工结论

既然当前只需要解析 `DXF`，那就不再为 `PDF`、`DWG` 和远端服务配置代理。实际派工压缩为：

- 1 个主代理负责共享入口、契约和集成
- 4 个第一批并行代理负责 DXF 基础、视图楼层片段、轴网、构件语义
- 1 个第二批代理负责拼接校核

这是当前仓库里既能并行、又不会在共享热点文件上互相打架的最小可执行分工。
