# 建筑自动建模需求转化文档

版本：`v0.1`

日期：`2026-03-30`

适用范围：`建筑自动建模系统中，从用户输入到 BIM 可执行指令的需求转化流程`

## 1. 文档目标

本文档单独定义“需求转化”这一核心能力，说明系统如何将用户的自然语言、图纸资料、表单参数和增量修改指令，转化为可执行的 `BIM` 建模命令。

本文档重点覆盖：

- 需求转化的定义与边界
- 输入与输出对象
- 转化流程分层
- 文本驱动与图纸驱动两类场景
- 自动补全与待确认项机制
- 构件筛选与批量替换机制
- 数据结构示例
- 风险、限制与质量指标

## 2. 需求转化的定义

本项目中的“需求转化”不是简单的文本解析，而是一条从“人类表达”到“机器可执行建模命令”的完整转换链路。

其本质为：

`用户需求 -> 结构化设计意图 -> 约束与假设 -> 执行计划 -> BIM 建模指令`

这里的转化对象包括：

- 用户一句自然语言描述
- 用户上传的 `CAD/DWG/DXF/PDF` 图纸
- 参数表单中的尺寸、层数、面积、风格等字段
- 模型生成后的增量修改指令

这里的转化结果不是普通文本，而是系统内部可执行的结构化对象，例如：

- `DesignIntent`
- `MissingFields`
- `Assumptions`
- `CompletionTrace`
- `ModelingPlan`
- `ModelPatch`
- `ElementSelector`

## 3. 目标与边界

### 3.1 核心目标

需求转化模块需要实现以下目标：

- 正确理解用户要建什么
- 区分硬约束、软约束和偏好项
- 在缺少信息时进行受控补全
- 将结果转换为统一 `DSL`
- 为参数化建模、规则求解和 `BIM` 装配提供稳定输入
- 支持首次生成与增量修改两种模式

### 3.2 设计边界

需求转化模块负责“理解与翻译”，但不直接负责：

- 最终几何求解
- `IFC` 装配
- 三维预览转换
- 最终人工审图责任

它是整个系统的中枢层，负责把输入变成可被后续模块执行的命令。

## 4. 输入分类

系统需支持四类输入：

### 4.1 自然语言输入

示例：

- “生成一栋 12 层住宅楼，两梯四户，现代风格”
- “我要一个小型办公楼，首层大堂挑高，南侧玻璃幕墙”

特点：

- 信息通常不完整
- 表达可能模糊或口语化
- 含有隐含默认前提

### 4.2 图纸资料输入

示例：

- `DWG` 平面图
- `DXF` 立面图
- `PDF` 剖面图
- 红线图、轴网图、功能分区图

特点：

- 信息量大，但格式不统一
- 图层命名和块定义可能混乱
- 需要将几何图元转成建筑语义对象

### 4.3 结构化表单输入

示例字段：

- 建筑类型
- 用地面积
- 层数
- 层高
- 容积率
- 建筑密度
- 户型配比
- 风格标签

特点：

- 可直接校验
- 与自然语言相比更适合做精确约束输入

### 4.4 增量修改输入

示例：

- “将南侧立面改为更简洁的玻璃幕墙”
- “标准层减少一户，增加共享会客区”
- “把模型内 800x1200 的窗全部更换为落地窗”

特点：

- 通常基于已有版本
- 需要识别修改范围
- 需要判断是否可局部重生成

## 5. 输出对象

需求转化模块的标准输出如下。

### 5.1 DesignIntent

表示经过归一化后的设计意图，是后续模块的主输入。

主要内容包括：

- 建筑类型
- 场地信息
- 层数与层高
- 功能与面积需求
- 风格偏好
- 输出格式

### 5.2 MissingFields

用于列出当前仍缺失但会影响建模结果的字段。

例如：

- 层高未指定
- 地区规则集未指定
- 首层是否架空未指定

### 5.3 Assumptions

用于记录系统自动补全的假设值。

例如：

- 住宅标准层层高默认 `3.0m`
- 办公建筑窗墙比默认 `0.5`

### 5.4 CompletionTrace

记录每个自动补全字段的来源，用于审计和回滚。

来源类型包括：

- 用户输入
- 图纸提取
- 模板默认值
- 规则库默认值
- `AI` 推断值

### 5.5 ModelingPlan

表示系统下一步要走的执行路径。

示例：

- 走文本驱动体量生成
- 走图纸驱动语义重建
- 仅执行门窗批量替换
- 触发局部重生成

### 5.6 ModelPatch

表示对已有模型的增量修改动作。

例如：

- 修改立面风格
- 替换门窗族
- 调整层高
- 新增阳台

## 6. 转化流程总览

需求转化应分为八个阶段。

### 6.1 输入接收与归档

系统统一接收：

- 文本
- 图纸
- 表单
- 修改指令

并为每次输入建立唯一上下文，绑定：

- `project_id`
- `version_id`
- `asset_id`
- `request_id`

### 6.2 内容抽取与标准化

对不同来源进行统一预处理：

- 文本进行分句、实体抽取和单位归一
- 图纸进行格式标准化、图层识别和图元抽取
- 表单进行字段校验
- 修改指令进行目标对象识别

标准化结果应消除以下差异：

- 米、毫米、厘米单位不一致
- “落地窗”与“通高窗”表达不一致
- “12 层”与“十二层”表达不一致

### 6.3 意图识别

该阶段负责回答三个问题：

- 用户要建什么
- 用户要改什么
- 用户最在意什么

意图类型建议分为：

- `create_building`
- `reconstruct_from_drawings`
- `modify_layout`
- `modify_facade`
- `replace_elements`
- `export_model`

### 6.4 约束分类

系统要把需求中的内容分成三类：

- `硬约束`：不能被违反，例如层数上限、红线退界、必须两梯四户
- `软约束`：尽量满足，例如更现代、更通透、更简洁
- `偏好项`：用于排序或风格选择，例如偏爱石材立面

该分类会直接影响后续求解策略。

### 6.5 缺失补全

对于缺失字段，系统按以下优先级补全：

1. 用户明确输入
2. 上传资料中可提取的信息
3. 建筑类型模板默认值
4. 地区规则库默认值
5. `AI` 推断

补全原则：

- 合规性相关字段不得静默补全
- 草案方案允许带假设运行
- 正式导出前必须提示待确认项

### 6.6 执行计划生成

系统根据当前意图和上下文，生成执行路径。

常见路径包括：

- `text_to_massing`
- `text_to_plan`
- `cad_to_bim`
- `element_batch_replace`
- `partial_regeneration`

### 6.7 BIM 指令生成

该阶段将 `DesignIntent` 或 `ModelPatch` 转为可被建模引擎执行的命令。

示例命令包括：

- 创建 `IfcBuildingStorey`
- 在指定轴网范围内创建 `IfcWall`
- 根据洞口参数创建 `IfcWindow`
- 按条件筛选窗构件并替换为新族

### 6.8 结果校验与回传

需求转化模块在输出前应附带：

- 置信度
- 缺失字段列表
- 自动补全项列表
- 建议澄清项
- 受影响范围

## 7. 文本驱动需求转化

### 7.1 场景定义

用户不上传图纸，只通过自然语言和少量表单来描述目标建筑。

### 7.2 转化重点

- 识别建筑类型
- 提取关键约束
- 自动补全常见默认值
- 判断是否足以进入体量生成

### 7.3 示例

用户输入：

“生成一栋 12 层住宅楼，两梯四户，现代风格，容积率 2.5。”

系统转化结果：

1. 识别主意图：`create_building`
2. 提取关键字段：住宅、12 层、两梯四户、现代、容积率 `2.5`
3. 发现缺失：地块边界、层高、户型面积配比、地区规则集
4. 自动补全：层高默认 `3.0m`
5. 标记待确认：地块、退界、限高
6. 输出执行路径：`text_to_massing -> text_to_plan -> bim_assembly`

## 8. 图纸驱动需求转化

### 8.1 场景定义

用户提供 `CAD/DWG/DXF/PDF` 图纸，希望系统先重建模型，再执行进一步修改。

### 8.2 转化重点

- 图纸语义识别
- 多视图配准
- 构件与空间重建
- 将修改指令映射为构件筛选条件

### 8.3 示例

用户输入：

- 上传住宅标准层 `CAD`
- 指令：“将模型内 800x1200 的窗全部更换为落地窗”

系统转化结果：

1. 识别主意图：`reconstruct_from_drawings + replace_elements`
2. 从图纸中提取墙线、窗块、门块、轴网、标注
3. 建立 `ParsedDrawingModel`
4. 将窗对象映射为 `IfcWindow`
5. 生成筛选条件：宽 `800mm`，高 `1200mm`
6. 生成动作：替换为 `floor_to_ceiling_window`
7. 输出执行路径：`cad_to_bim -> element_batch_replace -> validation`

## 9. 构件级需求转化

### 9.1 目标

支持用户通过自然语言指定某一类构件，并对其执行筛选、替换、删除、新增或属性修改。

### 9.2 构件筛选条件

建议支持以下筛选维度：

- 构件类型
- 宽度、高度、厚度
- 所在楼层
- 所在朝向
- 所属房间类型
- 材料类型
- 族名称
- 是否外墙构件

### 9.3 构件动作类型

- `replace_family`
- `update_property`
- `delete_element`
- `add_element`
- `move_element`

### 9.4 示例结构

```json
{
  "selector": {
    "ifc_type": "IfcWindow",
    "properties": {
      "overall_width_mm": 800,
      "overall_height_mm": 1200
    }
  },
  "action": {
    "type": "replace_family",
    "target_family": "floor_to_ceiling_window"
  },
  "scope": {
    "building_id": "bld_001"
  }
}
```

```json
    "convert_preview"
  ]
}
```

## 11. 澄清与待确认机制

并非所有缺失字段都应立即阻断流程。

建议分为两类：

### 11.1 可自动补全项

- 常见层高
- 默认窗墙比
- 常见交通核尺寸
- 常见门窗族

### 11.2 必须确认项

- 地块边界
- 地区规范集
- 限高
- 防火分区关键参数
- 涉及正式交付的法规性字段

系统输出时需要明确列出：

- 已自动补全的内容
- 待用户确认的内容
- 若用户不确认会带来的影响

## 12. 质量指标

需求转化模块建议重点监控以下指标：

- 意图识别准确率
- 结构化字段提取准确率
- 图纸语义识别准确率
- 自动补全接受率
- 构件筛选命中准确率
- 用户二次修正率
- 澄清请求触发率

## 13. 风险与限制

### 13.1 输入不规范

- 图纸图层混乱
- 尺寸标注缺失
- 文字描述过于模糊

### 13.2 补全风险

- `AI` 推断可能与用户真实意图不一致
- 默认参数可能不符合特定项目要求

### 13.3 语义映射风险

- 图纸中的块定义未必对应真实建筑构件
- 同一窗型可能存在多种命名方式

### 13.4 范围控制风险

- 用户一句修改可能引发大范围联动
- 构件替换可能影响立面、结构、栏杆和防火要求

## 14. 实施建议

建议按以下顺序建设需求转化能力：

### Phase 1

- 文本意图识别
- `DesignIntent` 结构化输出
- `MissingFields` 和 `Assumptions`
- 简单住宅场景补全策略

### Phase 2

- `CAD` 图纸标准化与语义识别
- 构件选择器
- 构件级批量替换
- 局部重生成范围判定

### Phase 3

- 多轮对话澄清
- 多规则集支持
- 企业标准模板接入
- 需求转化质量回放与评估平台

## 15. 结论

需求转化是整个建筑自动建模系统的中枢能力。

它决定了：

- 用户输入是否能被正确理解
- 模型是否具备可控性和可编辑性
- `AI` 是否真正服务于 `BIM` 建模，而不是生成无法落地的描述文本

如果后续系统要稳定运行，需求转化模块必须做到：

- 可解释
- 可审计
- 可回滚
- 可扩展
- 可与 `BIM` 引擎稳定衔接
