## 1. 会话目标概述

本次会话围绕“建筑类自动三维建模软件”的初步方案展开，核心目标是明确：

- 项目的总体定位

- 从 `CAD/PDF` 与文本需求到 `IFC` 输出的主链路
- `Revit` 在最终交付中的角色
- 各模块的职责、建议技术与使用方式
- 完整建模指令模板的设计方式

## 2. 本次已确认的核心结论

### 2.1 项目定位

项目定位已从“自动三维建模软件”逐步收敛为：

`面向建筑场景的方案级 BIM 自动建模系统`

其核心不是纯粹的 `text-to-3D`，而是：

`AI 需求理解 + 规则约束 + 参数化建模 + BIM 语义建模 + IFC 导出`

### 2.2 AI 与 BIM 的关系

本次已明确：

- 核心不是让 `AI` 直接操作某个 `BIM` 软件界面
- 核心是让 `AI` 将用户需求转化为结构化建模指令
- 真正的建模由程序化 `BIM` 引擎完成

结论：

`AI 驱动 BIM 建模系统`  
而不是  
`AI 直接操作 BIM 软件`

### 2.3 BIM 引擎定位

当前设计中推荐的核心 `BIM` 引擎为：

- `IfcOpenShell` 为主
- `OpenCASCADE` 为复杂几何可选增强

已明确：

- `IfcOpenShell` 负责程序化创建 `IFC/BIM` 语义对象
- `OpenCASCADE` 只在复杂布尔、异形构件、复杂实体几何时按需接入

### 2.4 输出与验证方式

本次已明确最终交付主链路为：

`输入 -> 解析 -> AI需求转化 -> 规则引擎 -> 参数化建模/执行计划 -> BIM引擎 -> 系统自检 -> 输出IFC -> 用户使用Revit查看结果`

其中：

- `系统自检` 是系统内部导出门禁
- `Revit` 不承担系统内部验证职责
- `Revit` 的角色是用户最终查看、确认与使用结果

### 2.5 输入范围

当前版本已收敛到：

- `CAD/DWG/DXF`
- `PDF`
- 文本需求

并明确：

- `DWG` 优先转换为 `DXF`
- 图纸解析优先走程序化解析，不依赖纯视觉模型
- 无图纸场景可走 `text_only` 的方案级 BIM 建模

## 4. 本次形成的模块化认知

### 4.1 图纸、文字信息接入

建议技术路线已明确为：

- `React`
- `FastAPI`
- `PostgreSQL`
- `MinIO / S3`

定位为：

- 前端输入与上传入口
- 原始图纸与文本信息归档
- 项目与版本上下文建立

### 4.2 CAD/PDF 解析

建议技术路线已明确为：

- `DWG TrueView`
- `ezdxf`
- `PyMuPDF`
- 可选 `Tesseract OCR`

定位为：

- `DWG -> DXF` 标准化
- `DXF` 图元、图层、标注解析
- `PDF` 文本和矢量信息提取
- 向 AI 和后续建模模块输出 `ParsedDrawingModel`

### 4.3 AI 需求转化

建议技术路线已明确为：

- `GPT-5.4`
- `GPT-5.4-nano`
- `Structured Outputs`
- `Function Calling`

定位为：

- 将用户需求和图纸语义转为结构化建模指令
- 产出 `DesignIntent / MissingFields / Assumptions / ModelPatch / ElementSelector`

### 4.4 规则引擎

建议技术路线已明确为：

- `Python`
- `YAML/JSON`
- `OR-Tools CP-SAT`

定位为：

- 管理硬约束和软约束
- 负责规则校验和复杂约束求解

### 4.5 参数化建模 / 执行计划

建议技术路线已明确为：

- `Python`
- `Pydantic`
- 自定义模板库

定位为：

- 决定全量重建还是局部重生成
- 决定走哪一条建模链路
- 将规则结果转为可执行计划

### 4.6 BIM 引擎

建议技术路线已明确为：

- `IfcOpenShell`
- 可选 `OpenCASCADE`

定位为：

- 程序化创建 `BIM` 语义模型
- 输出内部 `IFC` 语义对象

### 4.7 系统自检

建议技术路线已明确为：

- `IfcOpenShell`
- `IfcTester`
- `IfcDiff`
- 自定义校验器

定位为：

- IFC 结构校验
- 几何校验
- 业务规则校验
- 用户需求落实校验

### 4.8 自检不通过回退

建议技术路线已明确为：

- 任务编排器
- `Redis` 队列
- 自定义问题路由器

定位为：

- 根据问题类型回退到正确模块
- 不直接全量重跑

### 4.9 输出 IFC 文件

建议技术路线已明确为：

- `IfcOpenShell`
- 对象存储
- 数据库版本表

定位为：

- 只导出通过自检的正式 `IFC`
- 保存版本、路径与追溯关系

## 6. 本次识别出的关键限制

### 6.1 项目可做，但边界必须控制

已明确本项目存在以下限制：

- 图纸质量不统一
- 语义重建不可能一步到施工图级
- AI 补全不是事实，只能是可追溯的假设
- 法规和地区规范差异很大
- BIM 引擎只能执行结构化结果，不能替代理解与审图

### 6.2 GTJ 相关结论

本次已明确：

- `GTJ` 可作为后续生态对接目标
- 但不宜作为第一阶段核心原生输出目标
- 第一阶段更适合：

`CAD/PDF -> IFC -> Revit 查看/校核`

## 8. 会话中的关键判断

本次会话形成的关键判断如下：

- `IFC` 的生成逻辑应以程序化 `BIM` 引擎为核心
- `Revit` 不是建模核心，而是交付查看工具
- 模板系统必须前置，不然 AI 转化和规则校验都不稳定
- 第一阶段应优先实现开放 `IFC` 流程，而不是直接绑定私有生态格式
- 模块职责已经基本清晰，下一步适合进入字段、接口和规则字典层面
