# 新图纸未驱动模型检测报告

- **日期**: 2026-04-03
- **检测目标**: 核查“上传新的图纸后，生成的 IFC 模型仅楼层信息变化，其余主体仍与之前模型近似一致”的实际原因
- **结论**: 问题已确认，根因不在前端继续混入旧图纸，而在于这批新 DXF 图纸的主体几何大多以 `ACAD_PROXY_ENTITY` 形式存在，当前图纸解析器几乎未提取出可建模的 `wall / door / window / room` 实体，后续 BIM 链路因此退回 `template_fallback`。最终看到的模型只会随着表单楼层数变化而变化，主体构件仍沿用模板化输出。

## 一、检测范围

本次重点核查以下版本：

1. 基准版本
   - `proj_0021 / ver_0036`
   - 输入图纸：`专用宿舍楼-建施.dxf`、`专用宿舍楼-结施.dxf`
2. 问题版本
   - `proj_0023 / ver_0038`
   - 输入图纸：`二层平面图（不带家具）.dxf`、`一层平面图（不带家具）.dxf`
3. 复现版本
   - `proj_0026 / ver_0040`
   - 输入图纸：`二层平面图（不带家具）.dxf`、`一层平面图（不带家具）.dxf`

## 二、核心发现

### 发现 1：前端资产混入旧图纸不是这次问题的根因

数据库中的请求记录表明：

- `proj_0023 / req_0033` 的 `asset_ids` 只包含：
  - `asset_0041` → `二层平面图（不带家具）.dxf`
  - `asset_0042` → `一层平面图（不带家具）.dxf`
- `proj_0026 / req_0035` 的 `asset_ids` 只包含：
  - `asset_0045` → `二层平面图（不带家具）.dxf`
  - `asset_0046` → `一层平面图（不带家具）.dxf`

因此，这两次请求已经没有继续带入 `专用宿舍楼-建施.dxf` 或 `专用宿舍楼-结施.dxf`。之前的前端混图问题已经被隔离，这次属于新的后端解析/建模落模问题。

### 发现 2：新图纸已经被成功读取，但解析结果几乎为空

`proj_0026 / ver_0040` 的 `parsed_drawing` 摘要如下：

- `assets_count = 2`
- `asset_kinds = ["cad", "cad"]`
- `recognized_layers_count = 42`
- `detected_entities_total = 2`
- `detected_entities_emitted = 2`
- `space_candidates_detected = 0`
- `detected_entities` 仅包含：`site_boundary_candidate × 2`

这说明：

1. 文件不是没上传成功
2. DXF 不是没打开
3. 图层也不是没读到
4. 真正失败的是“从 DXF 图元中识别出可建模实体”这一层

### 发现 3：系统明确知道“墙体没有识别出来”

`proj_0023 / ver_0038` 与 `proj_0026 / ver_0040` 的规则问题完全一致，均包含：

- `drawing.review.wall_detection_low_confidence`
- `drawing.space_boundaries_missing`
- `drawing.review.site_boundary_inferred`

其中 `wall_detection_low_confidence` 的原始原因是：

`No wall lines were confidently detected; check layer mapping and source drawing conventions.`

这与最终只有 2 个 `site_boundary_candidate` 的解析结果相互印证。

### 发现 4：当前模型并不是按新图纸落模，而是退回模板输出

`proj_0026 / ver_0040` 的 BIM 元数据：

- `geometry_source = "template_fallback"`
- `source_wall_entities = 0`
- `source_window_entities = 0`
- `source_door_entities = 0`
- `source_storey_keys = []`

对应输出数量：

- `IfcWall = 16`
- `IfcDoor = 8`
- `IfcWindow = 16`
- `IfcSpace = 10`

这些构件都不是来自图纸，而是模板生成。也正因为如此，用户才会观察到：

- 楼层信息会变
- 但主体构件仍与之前模板化模型一致

### 发现 5：与基准图纸对比，差异非常明确

正常基准版本 `proj_0021 / ver_0036`：

- `geometry_source = "parsed_drawing"`
- `source_wall_entities = 2153`
- `source_storey_keys = ["1F", "2F", "RF"]`
- `IfcWall = 2153`
- `IfcWindow = 260`
- `IfcDoor = 12`
- `IfcSpace = 58`

问题版本 `proj_0026 / ver_0040`：

- `geometry_source = "template_fallback"`
- `source_wall_entities = 0`
- `source_storey_keys = []`
- `IfcWall = 16`
- `IfcWindow = 16`
- `IfcDoor = 8`
- `IfcSpace = 10`

这说明新图纸没有驱动主体构件建模，系统只是进入了模板兜底路径。

## 三、根因定位

### 直接根因：主体对象大量为 `ACAD_PROXY_ENTITY`

对 `proj_0026` 的两张新图纸逐层分析后发现：

#### 一层平面图（不带家具）

- 图元类型统计：
  - `ACAD_PROXY_ENTITY = 50`
  - `LINE = 7`
  - `LWPOLYLINE = 4`
- 关键图层分布：
  - `WALL -> ACAD_PROXY_ENTITY × 16`
  - `WINDOW -> ACAD_PROXY_ENTITY × 5`
  - `A-Wind -> ACAD_PROXY_ENTITY × 3`
  - `COLUMN -> ACAD_PROXY_ENTITY × 9`

#### 二层平面图（不带家具）

- 图元类型统计：
  - `ACAD_PROXY_ENTITY = 49`
  - `LINE = 27`
  - `LWPOLYLINE = 4`
- 关键图层分布：
  - `WALL -> ACAD_PROXY_ENTITY × 16`
  - `WINDOW -> ACAD_PROXY_ENTITY × 4`
  - `A-Wind -> ACAD_PROXY_ENTITY × 4`
  - `COLUMN -> ACAD_PROXY_ENTITY × 9`

### 机制根因：当前 `dxf_reader.py` 不处理代理实体

当前解析器 `app/parser/dxf_reader.py` 主要处理：

- `LINE`
- `LWPOLYLINE`
- `INSERT`
- `TEXT / MTEXT / ATTRIB / ATTDEF`
- `DIMENSION`

但对 `ACAD_PROXY_ENTITY` 没有相应展开或降级识别逻辑。

因此，虽然图层名里已经出现了明确的：

- `A-Wall`
- `WALL`
- `A-Wind`
- `WINDOW`
- `A-Spac-Iden`

解析器仍然无法把这些代理实体转成：

- `wall_line`
- `wall_path`
- `door_block`
- `window_block`
- `room_boundary`
- `room_label`

最终只保留了面积最大的闭合边界作为 `site_boundary_candidate`。

## 四、为什么会表现成“只改楼层，主体不变”

这是当前主链路的自然结果，不是偶发现象：

1. 新图纸进入系统
2. `drawing_parser` 能识别图层，但提不出主体构件
3. `BimEngine._source_layouts()` 得不到任何 `wall / door / window / room` 源实体
4. `geometry_source` 退回 `template_fallback`
5. IFC 按模板生成
6. 最终只有由表单/意图驱动的楼层数、层高等信息发生变化

所以表面现象就是：

- 换了图纸
- 楼层数可能跟着表单或推断结果改变
- 但主体墙门窗和空间还是模板化结果

## 五、风险判定

当前问题应判定为：**高优先级功能性缺陷**。

原因：

1. 系统表面上接受了新图纸并成功导出 IFC
2. 但主体模型并未真实反映该图纸
3. 用户容易误认为当前 IFC 是“按新图纸建出来的”
4. 这属于“静默退回模板”的伪成功，而不是显式失败

## 六、整改建议

### A. 解析层整改

优先级最高：

1. 为 `ACAD_PROXY_ENTITY` 增加识别策略
   - 若可展开为基础几何，则展开后进入现有 `wall/window/door/space` 分类
   - 若不可展开，至少在解析结果中输出“代理实体占比过高、无法建模”的明确阻断信号
2. 对 `A-Wall / WALL / A-Wind / WINDOW / A-Spac-Iden` 这类图层建立更强的代理实体兼容逻辑

### B. 门禁层整改

当前问题不应继续允许“静默模板落模”：

1. 当 `parsed.assets_count > 0` 且：
   - `source_wall_entities = 0`
   - `geometry_source = template_fallback`
   - 同时存在 `wall_detection_low_confidence`
2. 应新增正式门禁，阻止把这种结果误当作“图纸驱动模型”导出

建议新增类似问题码：

- `drawing.source_geometry_missing`
- `drawing.proxy_entities_unresolved`

### C. 展示层整改

前端结果页应明确显示：

- 当前版本是否为 `parsed_drawing` 还是 `template_fallback`
- 当使用模板兜底时，提示“当前 IFC 未按图纸主体几何生成”

## 七、最终结论

这次“上传新的图纸后，生成的 IFC 只是楼层信息变化，主体仍和之前模型一样”的问题已经确认。

**根因不是旧图纸继续混入请求，而是新图纸的主体对象主要以 `ACAD_PROXY_ENTITY` 形式存在，当前解析器未能提取可建模实体，导致 BIM 主链退回模板落模。**

因此当前系统对这批 `一层平面图（不带家具）.dxf / 二层平面图（不带家具）.dxf` 的输出，不能认定为“按图纸生成的模型”，只能认定为“读取图纸失败后退回模板的模型”。
