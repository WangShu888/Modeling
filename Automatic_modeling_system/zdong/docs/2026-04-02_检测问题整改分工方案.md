## 1. 目标

本轮整改不是只把链路“跑通”，而是达到以下交付口径：

- 能完整识别图纸中的有效建模信息，不再输出截断后的构件样本。
- 最终模型中的墙、门、窗、板、空间等核心构件，数量与位置和图纸解析结果一致。
- 最终交付必须满足“构件数量与位置正确、楼层按顺序拼接完整”的模型文件目标。
- 同一资产中的多楼层平面能够被切分、识别、按顺序拼接，最终导出完整 `model.ifc`。
- `ValidationReport` 成为正式放行门，不允许带着已知折叠或截断风险直接交付。

## 2. 当前检测出的全部问题

### 2.1 必修问题

1. `entity_detection_truncated`
   - 根因：`app/drawing_parser.py` 里的 `_MAX_DETECTED_ENTITIES = 200` 会导致复杂图纸候选构件被静默截断。
   - 直接影响：构件数量低估，后续 `IfcWall / IfcDoor / IfcWindow` 统计失真。

2. `multi_storey_asset_collapsed`
   - 根因：`app/storey_inference.py` 仍是 `asset_name -> storey_key` 单值模型，`app/pipeline.py` 的 `BimEngine._source_layouts()` 也按资产归组几何。
   - 直接影响：`1F / 2F / RF` 会折叠成一个楼层，无法可靠拼接。

### 2.2 联动修复问题

3. `drawing.space_boundaries_missing`
   - 根因：当前图纸解析没有稳定产出房间边界，BIM 空间退回模板兜底。
   - 直接影响：空间数量、空间面积、空间与构件关系不可信。

4. `site_boundary_inferred`
   - 根因：场地边界当前依赖推断，而非稳定识别。
   - 直接影响：总图相关约束、用地范围和后续校验可靠性不足。

5. `site.area_missing`
   - 根因：解析与意图层没有形成稳定的场地面积来源。
   - 直接影响：当前结果只能作为草案，不能作为正式交付口径。

### 2.3 已关闭问题

6. 真实样本路径写死
   - 状态：已在 `tests/test_dxf_delivery_pipeline.py` 修复。
   - 处理方式：不再作为本轮研发主任务，但必须保留回归测试。

## 3. 建议分工

建议按 4 条整改线并行，但执行顺序必须满足“先解析事实，再做楼层映射，再做 BIM 落模，最后由校验放行”。

| 整改线 | 主责组 | 主要文件/模块 | 必须解决的问题 | 交付物 |
| --- | --- | --- | --- | --- |
| A. 图纸解析增强 | 图纸解析组 | `app/drawing_parser.py` `app/models.py` `tests/test_drawing_parser_view_markers.py` `tests/test_dxf_delivery_pipeline.py` | `entity_detection_truncated` `drawing.space_boundaries_missing` `site_boundary_inferred` | 片段化 `ParsedDrawingModel`、完整构件候选、空间/场地识别统计 |
| B. 楼层切分与顺序拼接 | 图纸解析组 + BIM 执行组 | `app/storey_inference.py` `app/pipeline.py` `tests/test_storey_inference.py` `tests/test_bim_engine_source_geometry.py` | `multi_storey_asset_collapsed` | `asset fragment -> storey_key` 映射、按顺序的 `source_storey_keys`、可追踪的 `storey_layout_trace` |
| C. 构件落模与数量位置对齐 | BIM 执行组 | `app/pipeline.py` `app/ifc_runtime.py` `tests/test_bim_engine_source_geometry.py` `tests/test_validation_export.py` | 构件数量/位置偏差、空间退回模板的问题 | 构件级落模追踪、楼层内坐标保持、正式 `model.ifc` |
| D. 质量门禁与交付回归 | 规则与计划组 + 质量与回退组 | `app/planning.py` `app/pipeline.py` `tests/test_planning.py` `tests/test_validation_export.py` | 已知风险继续放行的问题 | 阻断规则、差异报告、真实样本回归门禁 |

## 4. 每条整改线的具体责任

### A 线：图纸解析增强

负责人：

- `agent_drawing_parser`

必须修改：

- `app/drawing_parser.py`
- 必要时扩展 `app/models.py`
- 解析层相关测试

任务清单：

1. 把构件识别从“硬截断”改成“可完整收集或可统计溢出量”的模式，至少要显式产出 `detected_entities_total`、`detected_entities_emitted`、`detected_entities_dropped`。
2. 为同一资产内多楼层视图补 `fragment_id / fragment_bbox / fragment_title / fragment_role`，切分依据采用“视图标题 + 图框/版面 + 几何聚类”联合策略。
3. 补齐房间边界和场地边界的识别统计，不允许只给一个总量而没有来源片段。
4. 按图层、按构件类型输出覆盖率摘要，方便下游确认“数量不对”究竟丢在解析还是落模阶段。

完成标准：

- 真实样本不再出现 `entity_detection_truncated`
- `ParsedDrawingModel` 能稳定区分 `1F / 2F / RF` 对应的视图片段
- 新增真实样本回归测试，覆盖多楼层切分和溢出统计

### B 线：楼层切分与顺序拼接

负责人：

- `agent_drawing_parser`
- `agent_bim_executor`

必须修改：

- `app/storey_inference.py`
- `app/pipeline.py`
- 楼层推断和 BIM 源几何测试

任务清单：

1. 把 `infer_parsed_asset_storeys()` 从 `asset_name -> storey_key` 改为 `fragment_id -> storey_key` 或等价的多值模型。
2. 在 `BimEngine._source_layouts()` 中按片段而不是按资产聚合墙、门、窗、房间。
3. 明确 `source_storey_keys` 的排序规则，只允许按 `storey_sort_key` 输出，不能靠输入顺序碰运气。
4. 为典型层复用、镜像层、屋面层增加显式来源字段，如 `source_fragment_id`、`source_storey_key`、`replicated`、`mirrored`。

完成标准：

- 真实样本中 `source_storey_keys` 至少覆盖检测到的显式楼层，并按顺序输出
- 不再出现 `multi_storey_asset_collapsed`
- `storey_layout_trace` 能追溯每个楼层来自哪个图纸片段

### C 线：构件落模与数量位置对齐

负责人：

- `agent_bim_executor`

必须修改：

- `app/pipeline.py`
- 必要时联动 `app/ifc_runtime.py`
- BIM 几何和导出测试

任务清单：

1. 基于片段后的源几何重建楼层内墙、门、窗、板、空间，不允许在片段缺失时直接用模板数量冒充真实数量。
2. 为每个导出构件保留最小追踪字段：`source_ref`、`source_fragment_id`、`source_storey_key`、局部坐标。
3. 增加“解析构件数 -> BIM 构件数 -> IFC 导出构件数”的对账摘要，至少覆盖 `IfcWall / IfcDoor / IfcWindow / IfcSlab / IfcSpace`。
4. 只有在片段事实完整的前提下，才允许使用典型层复制；复制行为必须在 `storey_layout_trace` 中可见。

完成标准：

- 核心构件数量与解析结果一致，或存在显式的可解释映射关系
- 构件楼层归属和局部坐标正确，不发生跨层串位
- 导出的 `model.ifc` 可被当前导出链路和标准 IFC 库读取

### D 线：质量门禁与交付回归

负责人：

- `agent_rules_planner`
- `agent_validation_delivery`

必须修改：

- `app/planning.py`
- `app/pipeline.py`
- 校验与导出测试

任务清单：

1. 把 `entity_detection_truncated`、`multi_storey_asset_collapsed` 从“可带告警放行”提升为“阻断正式导出”的门禁条件。
2. 新增三类校验：
   - 楼层完整性校验：解析楼层数、建模楼层数、导出楼层数一致。
   - 构件对账校验：解析层和 BIM 层的核心构件数量差异在阈值内。
   - 位置边界校验：楼层内构件坐标必须落在对应片段的包围盒内。
3. 对 `drawing.space_boundaries_missing`、`site_boundary_inferred`、`site.area_missing` 建立降级策略：
   - 若未补齐，不允许宣称“正式模型交付完成”。
4. 固化真实样本回归入口，要求每次发布前跑通 `上传 -> 解析 -> 建模 -> 校验 -> IFC 导出`。

完成标准：

- `ValidationReport` 成为真实放行门，而不是仅记录日志
- 正式导出前不再存在截断、折叠、未解释的数量差异
- 真实样本回归通过后，才允许输出最终 `model.ifc`

## 5. 排期与依赖

### 第一阶段：先消除伪正确

- A 线先完成溢出统计和片段切分骨架
- D 线同步把 `entity_detection_truncated`、`multi_storey_asset_collapsed` 设为阻断项

放行条件：

- 系统不能再把“截断后的 200 个候选”当成完整识别结果
- 系统不能再把“单图多楼层折叠为 1F”当成可交付模型

### 第二阶段：建立真实楼层模型

- B 线完成 `fragment -> storey` 映射和顺序拼接
- C 线开始基于片段几何重建楼层内构件

放行条件：

- `source_storey_keys` 顺序正确
- `storey_layout_trace` 可追溯
- 多楼层不再被资产级归并

### 第三阶段：形成正式交付

- C 线完成构件数量和位置对齐
- D 线完成对账校验和真实样本回归

放行条件：

- 核心构件数量、位置、楼层归属通过校验
- `ValidationReport` 无阻断项
- 导出结果包含完整 `model.ifc`

## 6. 最终验收口径

满足以下条件，才能判定“已经达到用户目标”：

1. 真实图纸中的多楼层视图都已被切分并建立楼层归属。
2. `source_storey_keys` 与 `storey_layout_trace` 能证明楼层是按顺序拼接完成的。
3. `IfcWall / IfcDoor / IfcWindow / IfcSlab / IfcSpace` 的数量与解析结果一致或有明确映射解释。
4. 关键构件坐标保留源图纸位置关系，不发生跨楼层、跨片段错位。
5. `entity_detection_truncated`、`multi_storey_asset_collapsed`、`drawing.space_boundaries_missing`、`site_boundary_inferred`、`site.area_missing` 不再以“未处理”状态进入正式交付。

## 7. 一句话执行口径

本轮整改不能再按“解析、建模、导出各修一点”推进，而要围绕两个主根因展开：先把图纸从“资产级”提升到“楼层片段级”，再让 BIM 和校验严格消费这些片段事实，最后才有可能交付构件数量正确、位置正确、楼层顺序完整的模型文件。
